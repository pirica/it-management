# Backfill missing audit triggers for 5 tables in `database.sql`

## Summary

Adds the missing INSERT/UPDATE/DELETE audit triggers for 5 tables that the static check `scripts/check_audit_logs_coverage.php` reported as uncovered. The triggers are added in `database.sql` (the schema source of truth), not applied via out-of-band UPDATEs to a live database — so any future fresh import provisions them automatically.

**Gap closed:** 5 tables × 3 events = **15 new triggers** in `database.sql`.

## Why

`scripts/check_audit_logs_coverage.php` compares every `CREATE TABLE` in `database.sql` against the `trg_{table}_audit_insert` triggers and reports any schema table that lacks the trigger. Before this patch, the script reported 5 missing tables:

| Table | Why it was missing |
|---|---|
| `bookmark_folders` | Bookmarks module (private/shared folder tree) |
| `bookmarks` | Bookmarks module (private/shared link list) |
| `password_entries` | Encrypted private password vault |
| `password_folders` | Vault folder hierarchy |
| `private_contacts` | User-scoped contacts (60 columns) |

These tables are mutated by user actions through the CRUD modules (`modules/bookmarks/`, `modules/bookmarks/bookmark_folders/`, `modules/passwords/`, `modules/private_contacts/`) and through `itm_log_audit()` / `itm_run_query()` calls. They were also modified directly via `INSERT`/`UPDATE`/`DELETE` from third-party flows (imports, admin actions). Without a database trigger, those direct mutations bypass PHP-level audit logging.

The fix lives in `database.sql` so the gap is closed at provision time and never re-opens on a fresh import.

## Files changed

- `database.sql` — 15 new triggers (5 tables × 3 events) appended at the end of the existing `DELIMITER $$` block, just before the final `DELIMITER ;`. No other tables, columns, or trigger bodies are touched.

## What was added

For each of the 5 tables, a complete trio of triggers was generated in the **exact existing template** (verified against `trg_companies_audit_*` and other working trios):

```sql
DROP TRIGGER IF EXISTS `trg_<table>_audit_insert`$$
CREATE TRIGGER `trg_<table>_audit_insert` AFTER INSERT ON `<table>` FOR EACH ROW BEGIN
  INSERT INTO `audit_logs` (`company_id`, `employee_id`, `actor_username`, `actor_email`,
                           `table_name`, `record_id`, `action`, `old_values`, `new_values`,
                           `ip_address`, `user_agent`)
  VALUES (COALESCE(@app_company_id, NEW.`company_id`, 0), @app_employee_id,
          @app_username, @app_email, '<table>', COALESCE(NEW.`id`, 0), 'INSERT', NULL,
          JSON_OBJECT('<col1>', NEW.`<col1>`, ..., '<colN>', NEW.`<colN>`),
          @app_ip_address, @app_user_agent);
END$$
```

…mirrored for UPDATE (with `OLD` and `NEW` objects) and DELETE (with `OLD` only).

The `DROP TRIGGER IF EXISTS …$$` line before each `CREATE TRIGGER` makes the patch idempotent — re-importing `database.sql` over an existing schema will not crash with "Trigger already exists".

The full column list is enumerated in `JSON_OBJECT(...)` for every table — including all 60 columns of `private_contacts` (TEXT, VARCHAR, LONGBLOB photo, etc.). The shape matches the existing triggers on large tables like `employees` and `equipment`.

## Verification

Run from the repository root with PHP 7.4.33 and MySQLi enabled:

```bash
php scripts/check_audit_logs_coverage.php
```

**Before this patch:** the script reported 5 missing tables and exited with code 2.

**After this patch:** the script reports 0 missing tables and exits with code 0. Total `trg_*_audit_insert` triggers = 123 (= 124 schema tables − 1 exempt `audit_logs`).

Additional static checks performed (without a local MySQL/PHP runtime):

- Replicated the script's regex analysis on the patched file: 0 missing tables.
- BEGIN/END block balance: 1 `BEGIN` per trigger, all closed by `END$$` inside the same `DELIMITER $$` wrapper.
- `DELIMITER $$` / `DELIMITER ;` markers preserved; final `DELIMITER ;` is the last non-empty line.
- JSON_OBJECT column-pair count for each new trigger matches the source `CREATE TABLE` column list (10 cols for `bookmark_folders`, 12 for `bookmarks`, 12 for `password_entries`, 8 for `password_folders`, 60 for `private_contacts`).
- `COALESCE(@app_company_id, NEW.\`company_id\`, 0)` pattern preserved on INSERT/UPDATE; `OLD` used on DELETE per the existing template.
- DROP TRIGGER count = 315 (was 300 + 15 new = correct: one DROP per new CREATE).

On a live MySQL VM, the patched `database.sql` can be re-imported and `check_audit_logs_coverage.php` will be re-run as the canonical gate.

## Out of scope

- No PHP module code changed.
- No `audit_logs` row data, no sample data, no FK relationships changed.
- No changes to the existing 354 triggers; the patch only adds new ones.
- No changes to `scripts/check_audit_logs_coverage.php` itself.
